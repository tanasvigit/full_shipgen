## 3) User Flows

### Login Flow
- User submits credentials on `/auth`.
- App checks 2FA requirement.
- If 2FA required → redirect `/auth/two-fa?token=...`.
- If verification required → create verification session → `/auth/verification`.
- Else authenticate and land on `/`.

### Logout Flow
- Triggered from console shell/header action.
- Session invalidated with loader.
- Redirect to auth route.

### 2FA Flow
- Validate temporary 2FA session.
- Submit OTP to verify.
- Success → authenticated session and redirect to console.
- Alternate actions: resend OTP, cancel/invalidate temp session.
- Additional policy surfaces in account/settings/admin pages.

### Invite / Join Flow
- User opens `/join/org/:public_id`.
- Submit invite code.
- Backend returns auth token/session.
- Optional forced set-password step.
- User lands in console.

### Console Navigation Flow
- Auth guard on `console` route.
- Sidebar/menu drives section navigation (`home`, `notifications`, `account`, `settings`, `admin`).
- Dynamic extension entries can inject extra menu items and virtual pages.

## 5) API Interaction Summary (High-Level)

### Bootstrap / App Init
- `GET installer/initialize` — installation/onboarding gate.

### Auth
- `POST auth/get-magic-reset-link` — forgot password.
- `GET auth/validate-verification` — validate verification session/token.
- `POST auth/reset-password` — set new password.
- `POST auth/create-verification-session` — prepare verification flow.
- `POST auth/validate-verification-session` — verify active verification session.
- `POST auth/verify-email` — complete verification.
- `POST two-fa/validate` — validate 2FA temp session.
- `POST two-fa/verify` — verify OTP.
- `POST two-fa/resend` — resend OTP.
- `POST two-fa/invalidate` — cancel 2FA temp session.

### Onboard / Invite / Install
- `POST onboard/verify-email`
- `POST onboard/send-verification-email`
- `POST onboard/send-verification-sms`
- `POST users/accept-company-invite`
- `POST users/set-password`
- `POST installer/createdb`
- `POST installer/migrate`
- `POST installer/seed`

### Console / Account / Settings
- Notifications (Ember Data + custom):
  - `GET notifications` (query/list)
  - `PUT notifications/mark-as-read`
  - `DELETE notifications/bulk-delete`
- Profile/org model saves (`user`, `company`) via adapter-backed updates (PATCH/PUT patterns).
- `GET lookup/timezones`
- `POST users/change-password`
- `GET/POST users/two-fa`
- `GET/POST companies/two-fa`
- `GET notifications/registry`
- `GET notifications/notifiables`
- `GET notifications/get-settings`
- `POST notifications/save-settings`
- Org actions:
  - `POST auth/create-organization`
  - `POST auth/join-organization`
  - `POST auth/switch-organization`

### Admin
- `GET settings/overview`
- `GET two-fa/config`
- `POST two-fa/config`
- `GET companies?view=admin...` (admin org listing query)
- `GET companies/:id/users`
- `POST auth/impersonate`
- `GET schedule-monitor/tasks`
- `GET schedule-monitor/:id`
- `GET schedule-monitor/:id/logs`
- Config modules use page/component-specific endpoints (some extension/internal-module owned).

## 6) State & Services Used (Rewrite-Critical)

- `session` — auth lifecycle, guards, logout, redirect memory.
- `currentUser` — user/company identity, role/admin context.
- `fetch` — HTTP client + upload handling.
- `store` — model query/save lifecycle.
- `router` — route transitions/refresh.
- `notifications` — toast/success/error UX messaging.
- `modalsManager` — confirmation/action modal workflows.
- `universe` — global event bus and dynamic behavior.
- `menu-service` — dynamic menu items and virtual-page resolution.
- `hook-service` — route lifecycle hook execution for extensions.
- `extension-manager` — extension bootstrap readiness.
- Onboarding services:
  - `onboarding-orchestrator`
  - `onboarding-registry`
  - `onboarding-context`
